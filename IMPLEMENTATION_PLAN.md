# Claude Code実装計画：デモポートフォリオサイト構築

前段の「デモポートフォリオ計画」をもとに、Claude Codeに実装させるための開発計画です。順序はポートフォリオサイト→デモ1→デモ2→デモ3→デモ4。GitHubへのアップロードを前提とし、リポジトリはPublic、ホスティングはGitHub Pagesとします（架空データのみを扱うため公開して問題ない前提）。UIデザインは各フェーズで`frontend-design@claude-plugins-official`プラグインに担当させます。事前にClaude Code側でこのプラグインをインストール・有効化しておいてください。

## 1. 全体方針・技術スタック

一人開発かつClaude Codeとの往復で進めることを踏まえ、バックエンドを持たない構成にします。ポートフォリオサイトと4つのデモはすべて1つのリポジトリ内の静的サイトとして実装し、計算ロジックはすべてクライアントサイドのTypeScriptで完結させます（架空データもリポジトリ内にJSONで同梱するため、サーバーやDBは不要です）。

技術スタックは、Vite + React + TypeScript + Tailwind CSSを採用します。ページ遷移はreact-router-domで、トップページ（ポートフォリオ）と `/demos/xxx` 配下の4デモページを構成します。Next.jsの静的書き出しでも実現できますが、GitHub Pagesとの相性・ビルドのシンプルさを優先してViteベースにしています。デモ4の可視化にはChart.js（react-chartjs-2）を使います。

UIデザインは`frontend-design@claude-plugins-official`プラグインに担当させ、最初のフェーズでデザインシステム（配色・タイポグラフィ・共通コンポーネント）を確立し、以降のフェーズはそれを再利用する形にします。ページ固有のビジュアルが必要な場面では、そのページに絞って再度このプラグインを使ってください。

## 2. リポジトリ構成

```
/repo
  .github/workflows/deploy.yml        # GitHub Pages自動デプロイ
  index.html
  vite.config.ts
  tailwind.config.js
  README.md
  CLAUDE.md                            # 運用ルール（架空データのみ使用、等）
  IMPLEMENTATION_PLAN.md               # このファイルをそのまま配置
  src/
    main.tsx
    router.tsx
    pages/
      Home.tsx                         # ポートフォリオトップ
      demos/
        ShoguKaizen/                   # デモ1：処遇改善加算 再計算・差異検出
        TeateTsugo/                    # デモ2：手当・勤怠突合
        KanshiShoko/                   # デモ3：計算根拠・監査証跡出力
        JinkenhiSim/                   # デモ4：人件費シミュレーション
    components/                        # frontend-design@claude-plugins-official の成果物（共通UI）
    lib/
      ruleEngine.ts                    # 共通の計算ロジック基盤
    data/                              # デモごとの架空データJSON
    assets/                            # 画像・動画サムネイル
  public/
```

## 3. フェーズ0：リポジトリ初期化・デザインシステム確立（目安1〜2日）

Vite + React + TypeScript + Tailwindでプロジェクトを初期化し、react-router-domで `/` と `/demos/:slug` の骨組みを作ります。GitHub PagesへのデプロイはGitHub Actions（`actions/upload-pages-artifact` + `actions/deploy-pages`）で自動化し、mainブランチへのpushをトリガーにします。SPAとしてGitHub Pagesで直接URLを開いても404にならないよう、`404.html`にindex.htmlの内容をコピーするSPAフォールバックの設定も入れておきます。

README.mdとCLAUDE.mdを作成し、CLAUDE.mdの先頭に「本リポジトリでは架空・匿名化データのみを使用し、実在の法人名・実データは一切含めない」というルールを明記します。ここまでできたら`frontend-design@claude-plugins-official`プラグインを使い、配色・タイポグラフィ・Button/Card/Table/Badge/Nav/Footerなど共通コンポーネントのデザインシステムを`src/components`配下に生成します。トーンは「Excel担当者が安心して使える、誠実で実務的な雰囲気」とし、派手なAI演出は避けるよう指示します。

完了条件は、ローカルでViteの開発サーバーが起動し、簡単なスタイルガイドページでデザインシステムのコンポーネントが確認でき、GitHub Actionsのビルド・デプロイが通ることです。

**Claude Codeへの依頼例**
「Vite+React+TypeScript+Tailwindでリポジトリを初期化し、react-router-domでルーティングの骨組みを作ってください。GitHub Pages向けのGitHub Actionsデプロイワークフローも設定し、SPAの直リンク・リロード対策として404.htmlのフォールバックも入れてください。README.mdとCLAUDE.md（架空データのみ使用する旨を明記）も作成してください。その上で`frontend-design@claude-plugins-official`プラグインを使い、介護福祉法人向け業務効率化サービスのポートフォリオサイト用デザインシステムを作ってください。」

## 4. フェーズ1：ポートフォリオサイト（目安3〜4日）

トップページを実装します。構成はサービス概要、顧客の課題全体像、4デモへの導線カード、無料ヒアリングへのCTAです。各デモの個別ページはこの時点では中身を作り込まず、「準備中」のプレースホルダーとして枠だけ用意し、後続フェーズで埋めていきます。問い合わせ導線はGoogleフォームへの外部リンクで十分です。

完了条件は、トップページと4つのプレースホルダーデモページがGitHub Pagesで公開され、相互に遷移できることです。

**Claude Codeへの依頼例**
「フェーズ0のデザインシステムを使ってトップページを実装してください。構成は［デモポートフォリオ計画.mdのポートフォリオサイト計画セクションを貼り付け］の通りです。4デモへのカードは中身がまだないので『準備中』のプレースホルダーページだけ作ってください。」

## 5. フェーズ2：デモ1（処遇改善加算 再計算・差異検出、目安5〜7日）

架空50名分のダミーデータ（施設・職員属性・資格・勤続年数・現行の加算配分額）を`src/data/demo1.json`として設計します。加算配分ルール（キャリアパス要件・月額賃金改善要件に基づく配分ロジック）を`src/lib/ruleEngine.ts`に実装し、単体テストを書きます。UIは、データ選択→計算結果テーブル→現行結果との差異ハイライト→職員別の計算根拠パネル、という流れにします。差異が出た職員には理由候補を表示し、判断できないケースは計算を止めて保留リストに出す設計にします。仕上げにデモ動画・GIFを撮影し、トップページのデモ1カードにリンクさせます。

完了条件は、デモ1ページが単独で動作し、差異検出結果と計算根拠が確認できることです。

## 6. フェーズ3：デモ2（手当・勤怠突合、目安4〜5日）

フォーマットの異なる3施設分のダミーデータを用意し、列名・項目のゆらぎを吸収する自動マッピングロジックを`ruleEngine.ts`に追加します。そのうえで夜勤手当・資格手当・兼務按分の計算ロジックを実装し、UIは施設別データの取り込み画面から統合結果画面への流れにします。デモ1の計算エンジンを再利用することで開発量を抑えます。

完了条件は、デモ2ページが単独で動作し、施設をまたいだ統合結果と按分計算が表示されることです。

## 7. フェーズ4：デモ3（計算根拠・監査証跡自動出力、目安2〜3日）

デモ1・デモ2の計算ロジックに監査ログ機能を追加し、入力値・適用ルール・計算過程・修正履歴を職員単位で追跡できるようにします。PDFまたはExcel形式でのエクスポート機能を実装し、UIは監査ログビューアとしてまとめます。既存ロジックへの追加なので開発量は小さめです。

完了条件は、職員単位の計算履歴がエクスポートでき、担当者の引き継ぎ資料としてそのまま使える体裁になっていることです。

## 8. フェーズ5：デモ4（人件費シミュレーション・経営レポート、目安3〜4日）

デモ1のデータを土台に、加算率の変更や採用計画（増員・退職）をシナリオとして入力できるUIを作り、人件費総額への影響をシミュレーションします。可視化はreact-chartjs-2で折れ線・棒グラフを表示します。

完了条件は、シナリオを変更するとグラフがリアルタイムで更新されることです。

## 9. フェーズ6：統合・公開・運用（目安1〜2日）

全デモページの相互リンクとトップページのCTAを最終確認し、レスポンシブ対応とアクセシビリティ（コントラスト、キーボード操作）をチェックします。README.mdを最終化し、必要ならカスタムドメインを設定して本公開します。

## 10. GitHub運用ルール

mainブランチは常にデプロイ可能な状態を保ちます。フェーズごとに`feature/portfolio-site`、`feature/demo1-shogu-kaizen`のようなブランチを切り、PRでmainにマージします。フェーズ単位でGitHub Issueを立てておくと、Claude Codeに「このIssueを実装して」と渡しやすく、1フェーズ=1Issue=1PRで進めると差分のレビューもしやすくなります。GitHub Actionsは、push時にlintとビルドの確認を行い、mainへのマージ時にGitHub Pagesへ自動デプロイする2段構成にします。

## 11. Claude Codeへの引き継ぎ方

このファイルをリポジトリ直下に`IMPLEMENTATION_PLAN.md`として置き、CLAUDE.mdからも参照させておくと、Claude Codeが毎回全体像を踏まえて作業できます。1フェーズ=1セッション（1PR）で依頼すると、コンテキストが散らからずレビューもしやすくなります。UI実装のたびに「フェーズ0で`frontend-design@claude-plugins-official`が作ったコンポーネントを使ってください」と明示すると、デザインの一貫性が保たれます。

## 12. データ・セキュリティ上の注意

Publicリポジトリで運用するため、架空・匿名化データ以外は一切コミットしないことを徹底してください。`.gitignore`に実データ用のファイル名パターン（例：`*_real.csv`、`/private-data/`）をあらかじめ入れておくと、うっかりコミットの事故を防げます。GitHub Pagesは静的ホスティングのみで、サーバーサイドでの外部送信は発生しません。問い合わせフォームは外部サービス（Googleフォーム等）へのリンクまたは埋め込みにとどめ、この段階では個人情報を自前のサーバーに送信する実装は行いません。将来、実際の顧客データを使った実証やフォーム経由での個人情報収集機能を追加する場合は、その時点でデータの保存場所・送信先・セキュリティレベルを改めて確認してから着手してください。
